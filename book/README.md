# Издательский контур книги «Пуэр. Живая гора»

Работа над книгой ведётся в фиксированной последовательности:

1. Валидировать исходные данные командой `npm run book:validate:data`.
2. Подготовить черновик, отмечая проверяемые утверждения маркерами источников (`claim markers`).
3. Закрыть редакционные, фактологические и экспертные рецензии.
4. Получить и зафиксировать права на все тексты, изображения, карты и другие материалы.
5. Собрать редакционную пробу (`proof`).
6. Отрендерить пробу в постраничный формат.
7. Визуально проверить страницы: композицию, переносы, подписи, изображения, поля и вылеты.
8. Получить у выбранной типографии актуальные технические требования.
9. Экспортировать финальные файлы печати в требуемом типографией стандарте PDF/X.
10. Выполнить предпечатную проверку (`preflight`) экспортированных PDF/X.

Редакционная проба и финальный печатный PDF/X — разные артефакты. Проба предназначена для согласования содержания и визуальной проверки и не считается готовым файлом для типографии. Финальный PDF/X создаётся только после закрытия рецензий и прав, получения требований типографии и исправления замечаний к пробе; именно экспортированный PDF/X проходит итоговый preflight перед передачей в печать.

## Режимы proof-PDF

Редакционная проба сохраняет служебные роли полос, связи тезисов с ключами источников, статусы визуальных материалов и редакционные placeholders. Это вход frozen workflow внешней рецензии; команда по умолчанию и имена её файлов не изменились:

```bash
book/.venv/bin/python book/scripts/build_proof.py --mode editorial
```

Она создаёт `book/output/pdf/puer-album-proof.pdf` и `book/output/pdf/puer-guide-proof.pdf`.

Читательская проба предназначена только для проверки читаемости без внутреннего производственного аппарата:

```bash
book/.venv/bin/python book/scripts/build_proof.py --mode reader
```

Она создаёт отдельные `book/output/pdf/puer-album-reader-proof.pdf` и `book/output/pdf/puer-guide-reader-proof.pdf`, не заменяя редакционные PDF. Читательская проба всё ещё не является финальным художественным оформлением, готовым файлом для печати или PDF/X. Она не подтверждает закрытие прав на изображения и не означает получения независимого экспертного одобрения.

## Пакет внешней рецензии

После успешного proof-preflight параметры цикла фиксируются в `book/config/review-cycle.json`. Команда:

```bash
npm run book:review:prepare
npm run book:review:verify
```

Первая команда проверяет frozen commit, валидность рукописи, SHA-256 и внутреннюю структуру обоих локальных proof-PDF, затем создаёт неизменяемый пакет в `book/output/review/<cycle-id>/`. Вторая команда повторно и без записи проверяет точный состав пакета, все хеши, snapshot, три ролевых запроса, пустоту response templates и оба PDF. Каталог `book/output/` не входит в Git: при передаче рецензенту к пакету нужно приложить оба PDF, перечисленные в `manifest.json`, и сверить их хеши.

Каждая из трёх ролей получает одинаковый корпус из 70 тезисов, полную библиографию и отдельную профильную выборку. Каждый напечатанный `claim marker` входит в корпус, если тезис не отклонён; дополнительно включаются зарегистрированные тезисы со статусом `checked` или `verified`, даже если они обозначены как registry-only. Ненапечатанный тезис со статусом `draft` автоматически не включается. Это fail-closed правило гарантирует структурированное решение историка, чайного технолога и медицинского специалиста по каждому тезису корпуса. Ответ `not-in-scope` одобрением не считается.

До отправки необходимо подтвердить предлагаемый дедлайн и указать реальных рецензентов с проверяемой квалификацией. Пустые response templates не являются одобрением. Текущее состояние фиксируют `book/production/review-dispatch.json` и `book/production/review-closeout.md`; пока внешние ответы не получены, они обязаны сохранять статусы `prepared-not-dispatched` и `NOT CLOSED`.

Статус `approved` проходит data-validator только при совпадении cycle/proof/snapshot, подтверждённом дедлайне и отправленном цикле, наличии проверяемых данных эксперта, COI и финансирования, допустимого списка просмотренных полос, временных меток и SHA-256 фактически сохранённого ответа. Для просроченного ответа нужен явный редакционный waiver; для отклонённого тезиса используется отдельное решение `confirm-exclusion` без вымышленных proof-страниц. На стадии подготовки evidence context отсутствует, поэтому любой `approved` закрывается fail-closed. Контролируемый импорт и архив ответов выполняются отдельным шагом до closeout. Это технический stop-gate, а не автоматическая проверка личности: квалификацию и принадлежность URL реальному эксперту редакция подтверждает отдельно.

Любое изменение рукописи, данных или proof-PDF создаёт новый цикл с новыми хешами. Одобрение предыдущего цикла автоматически не переносится.
